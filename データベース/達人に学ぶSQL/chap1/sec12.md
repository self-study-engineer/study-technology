#### テーブル設計

##### 名前と意味

人間というのは総じて「無意味なもの」や「規則性のないもの」を扱う能力が低い特徴を持っており、精神的に悪影響を及ぼすことも知られている。<u>RDBでは「アドレス」という無意味なものを排除し、テーブル名やカラム名などの「名前」が残っている</u>。名前をつける際に標準SQLで許可されている文字は以下の3つあり、一部のDBMSでは特殊文字(`$, #, @`など)も許可されているが基本的には避けるべきである。

1. アルファベット
2. 数字
3. アンダーバー(_)

##### 属性と列

RDBは「位置情報」はなく「名前」を管理しているため、以下のような利用方法は御法度とされている。

- 「年収」を意味していた列に「収めた税金」が格納されている運用(様式変更に伴って発生するような事象)
- EAV(エンティティ・アトリビュート・バリュー)や単一参照テーブルなどの「位置によるデータ呼び出し」

<font color=red>DBにおいて列とはある実体の「属性」であり永続的な存在としてみなされるため、アプリケーション層で利用される一時的な変数とは異なる</font>。そのため、ある時は年齢、ある時は体重、ある時は...、のように**列の意味が時と場合によって七変化するような設計はやらないようにすることが肝要である**。

#### コーディングの指針

##### コメント

コメントはコーディングスタイルの中でも特に議論の的になるテーマであり、「**絶対にコメントはつけるべき**」という人もいれば、「**コードの可読性を下げるからコメントがなくても読めるコーディングをするべき**」という人もいる。開発メンバ間(デザイナー、プログラマー、テスターなど)、技術領域別(UI層、アプリケーション層、インフラ層など)でそれぞれで考え方は異なるはずであるが、SQLにおいてはコメントはあった方が良い。理由は以下の2つ。

- 【**理由1**】SQLは宣言型言語であり手続き型言語に比べてコードに多くの処理を詰め込むため、「語らせる」コードを書くことが難しいから。
- 【**理由2**】<font color=red>SQLでは段階的な実行デバッグがほとんどできず、コード解析時に机上デバッグになることが多い</font>から。

SQLにおけるコメントの書き方は2種類あり、以下に示す。

```sql
-- 1行コメント
-- SomeTableからcol_1を選択するよ
SELECT col_1 FROM SomeTable;

/*
複数行コメント
SomeTableからcol_1を選択するよ
*/
SELECT col_1 FROM SomeTable;
```

##### インデント

以下にインデントだけが異なるクエリとインデントを入れる考え方をそれぞれ示す。可読性が明らかに異なることがわかる。

- サブクエリは「サブ(下位)」という接頭語がつくことからわかるように意味的にも1段階インデントを下げて書いても良い。
- `SELECT`や`GROUP BY`において列を複数指定する場合はこれ1段階インデントを下げ、「句」の区切りを明確にしても良い。

```sql
-- ○ 良い例
SELECT col_1, 
       col_2, 
       col_3, 
       COUNT(*)
  FROM tbl_A
 WHERE col_1 = 'a'
   AND col_2 = (SELECT MAX(col_2)
                FROM tbl_B
                WHERE col_3 = 100) 
 GROUP BY col_1, 
          col_2, 
          col_3;

-- × 悪い例
SELECT col_1, col_2, col_3, COUNT(*)
FROM tbl_A
WHERE col_1 = 'a'
AND col_2 = (
SELECT MAX(col_2)
FROM tbl_B
WHERE col_3 = 100
) GROUP BY col_1, col_2, col_3;
```

<div style="page-break-before:always"></div>

また、別の視点としてキーワードの「先頭を合わせるケース」と「末尾を合わせるケース」の視点もある。ここまでくると好き嫌いの領域を出ないが、こういった視点があることも頭の片隅に入れておくと良い。

```sql
-- 先頭の位置を揃えるケース
SELECT 
FROM
WHERE
GROUP BY
HAVING
ORDER BY

-- 先頭の位置を揃えるケース
SELECT 
  FROM
 WHERE
 GROUP BY
HAVING
 ORDER BY
```

##### スペース

どんな言語においてもコードには適切なスキマが必要である。コードを詰めて書くと、意味的な単位が不明確になり、解読に余計な労力が発生する。そのため、<u>要素を明示的に区切り、人の目から見て読みやすい心がけが必要である</u>。

```sql
-- ○ 良い例
SELECT col_1
  FROM tbl_A A INNER JOIN tbl_B B
    ON A.col_3 = B.col_3
 WHERE ( A.col_1 >= 100 OR A.col_2 IN( 'A', 'B' ) );

-- × 悪い例
SELECT col_1
  FROM tbl_A A INNER JOIN tbl_B B
    ON A.col_3=B.col_3
 WHERE (A.col_1>=100 OR A.col_2 IN('A','B'));
```

#### 大文字と小文字

一般によく使われる命名規則のうち以下の3つ。

- 【**スネークケース**`snake_case`】単語をアンダースコア（_）で区切る形式。Python、Ruby、C言語など。
- 【**パスカルケース**`PascalCase`】すべての単語の先頭文字を大文字にする形式。C#、Delphi、Javaなどのクラス名やタイプ名。
- 【**キャメルケース**`camelCase1`】最初の単語は小文字、続く単語の先頭文字を大文字にする形式。JavaScript、Java、C#など。

SQLにおいて大文字・小文字の使い分けはかなり共通理解が成立しており、<font color=red>予約語は大文字、列名やテーブル名は小文字(要素後の頭文字だけは大文字を使う流派もある)、ということになっている</font>。

```sql
-- 予約後(SELECT, FROMなど)は「大文字」
-- 列名やテーブル名は「小文字」
SELECT col_1, col_2
  FROM tble_A
 WHERE col_1 = 'a'
 GROUP BY col_1, col_2;
```

##### カンマ

カンマは要素の「後ろ」に置くと考える人が多いが、別の言い方もできる。それが以下の考え方がである。

$$
カンマは要素と要素の中間に置く
$$

この考え方を念頭に置くと、以下の書き方の発想がわかる。

```sql
-- カンマの位置について
SELECT col_1
      ,col_2
      ,col_3
      ,col_4
FROM tbl_A
```

上記の書き方のメリットとデメリットは以下の通り。

- 【**メリット**】
  - `col_2`〜`col_4`を削除してもSQLエラーにならない
  - 矩形選択の機能を持つエディタでの編集がしやすい
- 【**デメリット**】
  - 可読性を下げる

##### ワイルドカードは使わない

ワイルドカード(`*`)で全列を指定すると、テーブルの全列を選択するが、以下のデメリットがある。

- 【**デメリット**】
  - 論理的に不要な列まで含まれる
  - 可読性を低下し仕様変更にも弱くなる
  - 列の順番を入れ替えたり、追加・削除が発生したときに結果が狂う原因となる

```sql
-- ×
SELECT * FROM SomeTable

-- ○ 
SELECT col_1, col_2, col_3, ... FROM SomeTable
```

<u>原則としては列指定を行い、数百行も保持するようなテーブルにおいてはケースバイケースで対応するというのが現実的な運用になる</u>。

##### 標準語を話そう

SQLは数ある言語の中でも方言が多い方であり、言語によって独自の拡張がされている。特に気をつけるべきポイントは以下の2つ。

1. 【**実装依存の関数・演算子を使わない**】
DECODE(Oracle)、IF(MySQL)、NVL(Oracle)、STUFF(SQL Server)等の関数があり、それらは基本的には使わない方が無難である。<font color=red>CASE式やCOALESCE、NULLIFなどの標準関数を使い、移植性が高い状態を保つことを心がけるべきである</font>。
2. 【**結合には標準の構文を使う**】
SQLの文法で最も実装依存性が高いものが結合構文である。<font color=red>標準SQLでは`INNER`や`CROSS`、`LEFT OUTER`、`RIGHT OUTER`などの結合の種類を示すキーワードを結合条件も`ON`句に分離して書くことになっているため、それに準拠して書くべきである</font>。(+)演算子(Oracle)や*=演算子(SQL Server)などの実装依存の書式は移植性を下げるため避けるべきである。

##### 外部結合の「左派」と「右派」

外部結合には、左、右、完全の3種類があり、特に左と右の表現力は同じため、論理的にはどちらを使っても良いことになる。人は「左→右でものを見る」ため、基本は「左部外部結合」を使用すれば良い。

##### 相関サブクエリを追放せよ

昔からSQLは<b>①NULL、②量化、③相関サブクエリ</b>の3つが挫折のポイントとして言われている。そのうち、③相関サブクエリはウィンドウ関数を用いて消去することができている(WinMagic)。

##### FROM句から書く

「皆さんはSQLを書くとき、どの句から書き始めますか？」圧倒的に「SELECT句」だと思います。10行程度のSQLであればどこから書き始めても良いと思いますが、SQLが大きく複雑になればなるほど「**SELECT句から書き始める方法**」はわかりづらくなると思います。その理由はSQLの実行順序にあります。SQLにおいて、`FROM`句が最初に実行され、次いで`JOIN`、`WHERE`と処理されていきます。そのため、SQLを書くときは「**FROM句から書き始める方法**」もあるということになります。

$$
FROM→JOIN→WHERE→GROUP\hspace{1mm}BY→HAVING→SELECT→...
$$
